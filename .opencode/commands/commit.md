Выполни фиксацию изменений в git-репозитории. **Режим: $1** (если аргумент "push" — выполнить push после коммита, иначе только локальный коммит).

## Шаги

### 1. Сбор информации
Получи состояние репозитория:
- !`git status`
- !`git diff --stat`
- !`git diff --cached --stat`
- !`git log --oneline -5`

### 2. Анализ изменений
- Посмотри `git diff` и `git diff --cached` чтобы понять что изменено
- Если есть untracked файлы — посмотри их содержимое при необходимости
- Определи характер изменений: новый функционал, багфикс, рефакторинг, документация и т.д.

### 3. Проверки перед коммитом
- **Секреты**: Убедись что в staged файлах нет секретов (.env, credentials, API ключей, паролей). Если нашёл — предупреди и НЕ коммить.
- **Фронтенд**: Запусти `bash check_frontend.sh` из корня проекта. Если проверки не прошли — НЕ коммить, покажи ошибки пользователю.
- **Features**: Если изменения добавляют новую функциональность — напомни обновить docs/features.md

### 4. Формирование сообщения коммита
Сгенерируй сообщение в формате conventional commits:
- `feat: ...` — новый функционал
- `fix: ...` — исправление бага
- `refactor: ...` — рефакторинг
- `docs: ...` — документация
- `chore: ...` — технические изменения
- `style: ...` — форматирование
- `test: ...` — тесты
- `ci: ...` — CI/CD

Сообщение должно:
- Быть кратким (1 строка, до 72 символов)
- Отражать СУТЬ изменений, а не перечислять файлы
- Быть на английском языке

### 5. Выполнение коммита
1. `git add` все релевантные изменённые файлы (НЕ добавляй .env, secrets, node_modules, __pycache__ и т.д.)
2. `git commit -m "сообщение"`
3. Проверь `git status` после коммита

### 6. Push (если режим push)
Если первый аргумент = "push":
1. Проверь что коммит прошёл успешно
2. Определи текущую ветку: !`git branch --show-current`
3. Выполни `git push` (или `git push -u origin <branch>` если ветка не отслеживается)
4. Если push отклонён — НЕ делай force push, предупреди пользователя

### 7. Итог
Выведи краткий результат:
- Что было закоммичено (сообщение коммита)
- Сколько файлов изменено
- Push выполнен или нет

## Важно
- НИКОГДА не делай `git push --force`
- НИКОГДА не коммить секреты
- НЕ обновляй git config
- НЕ используй --no-verify
- НЕ делай git commit --amend
