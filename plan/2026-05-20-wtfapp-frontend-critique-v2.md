## Шаг 1 — Пять линз критики (Прохождение 2)

---

### Линз 1: Полнота

**C2-1.1** 🟡 **Бэкенд возвращает `refresh_token` в JSON-ответе (строка 259 бэкенд-дизайна), а фронтенд ожидает `Set-Cookie`.**

Бэкенд: `POST /api/v1/auth/login` — «Вход (возвращает access + refresh)». Это означает `refresh_token` возвращается в теле ответа. Но фронтенд v2 (строка 284) ожидает: `Set-Cookie: refresh_token (httpOnly, SameSite=Strict, Path=/api/v1/auth)`.

Это **фундаментальное рассогласование архитектуры auth** между фронтендом и бэкендом. Либо:
- Бэкенд должен быть изменён: не возвращать refresh_token в JSON, а ставить httpOnly cookie через `Set-Cookie` заголовок. Это требует правок бэкенд-дизайна.
- Либо фронтенд должен вернуться к localStorage для refresh_token, что возвращает XSS-риск.

Фронтенд-план не может быть реализован без соответствующего изменения бэкенда. Это нужно явно зафиксировать как зависимость.

**C2-1.2** 🟡 **Нет описания поведения при неудачном refresh при старте приложения.**

Строка 293: `Неудача → пользователь не авторизован (без redirect)`. Но что если пользователь был на защищённой странице (`/profile`), перезагрузил страницу, и refresh не удался? Он увидит `/profile` без данных. Нужен редирект на `/login` для защищённых роутов при неудачном refresh, а не «без redirect».

**C2-1.3** 🟡 **Нет описания `NearbyFilters` типа.**

`mapStore.filters: NearbyFilters` (строка 523) — тип упомянут, но не определён. Какие именно поля он содержит? Это критический тип — он используется в FilterPanel и в `useNearbyToilets`. Нужен интерфейс.

**C2-1.4** 🟡 **Нет описания API-ответа для `GET /api/v1/toilets/{id}` (детали туалета).**

Для `/nearby` бэкенд описывает формат ответа. Но для `GET /toilets/{id}` — нет ни в бэкенд-дизайне, ни во фронтенд-дизайне. ToiletPage рендерит: название, адрес, этаж, статус, рейтинг, бейджи, часы, описание, отзывы, фото. Но какой именно JSON приходит? Это нужно для корректной типизации.

**C2-1.5** 🟡 **Нет описания формы загрузки фото (multipart/form-data).**

Бэкенд: `POST /api/v1/toilets/{id}/photos` — загрузка фото. Но не указано: один файл за запрос или массив? `Content-Type: multipart/form-data`? Какое имя поля (`file`? `photo`? `photos`?)? Это нужно для реализации PhotoUploader.

**C2-1.6** 🟢 **Нет описания кастомных иконок маркеров ToiletMarker.**

Строка 109: «Иконка зависит от `toilet_type`» + цвет по рейтингу. Нужны 3 иконки × 4 цвета = 12 вариантов. Это SVG-иконки? PNG-спрайт? L.divIcon? Кастомные L.icon? От выбора зависит подход к реализации и производительность.

### Линз 2: Согласованность

**C2-2.1** 🔴 **httpOnly cookie auth требует изменения бэкенд-дизайна — это не зафиксировано.**

(Дублёр C2-1.1, но через линзу согласованности.) Фронтенд v2 спроектирован вокруг httpOnly cookie для refresh_token, но бэкенд-дизайн остаётся на возврате refresh_token в JSON-ответе. Два документа рассогласованы по критически важному архитектурному решению. Нужен либо:
- Явный раздел во фронтенд-дизайне: «Требования к бэкенду: login/register должны устанавливать refresh_token как httpOnly cookie вместо возврата в JSON»
- Или синхронное обновление бэкенд-дизайна

**C2-2.2** 🟡 **ForgotPasswordPage описана как Priority 2, но в роутах и структуре проекта она присутствует.**

Строка 53: роут `/forgot-password` есть. Строка 224-227: ForgotPasswordPage описана детально. Строка 772: «ForgotPasswordPage (полный flow)» — Priority 2. Противоречие: если роут и страница описаны, значит ли это что MVP включает заглушку? Или полная реализация?

**C2-2.3** 🟡 **`paper_type` в FilterPanel: значение «Не важно» не является API-параметром.**

Строка 152: FilterPanel показывает «Бумага: [Не важно] [Нет] [В кабинке] [В диспенсере] [Оба]». Но бэкенд-параметр `paper_type` (строка 289 бэкенда) принимает значения `none / in_cabin / dispenser / both`. «Не важно» = не передавать параметр. Это корректно, но стоит явно указать: «Не важно» → параметр не включается в запрос.

**C2-2.4** 🟡 **`has_water` в FilterPanel: «вода (горячая/холодная)» — но API имеет `has_water` и `has_hot_water` как отдельные параметры.**

Строка 151: FilterPanel описывает «вода (горячая/холодная)» как одну позицию. Но бэкенд имеет два отдельных фильтра: `has_water` и `has_hot_water`. Нужно ли два отдельных toggle? Или «вода» = `has_water`, «горячая вода» = `has_hot_water`?

**C2-2.5** 🟡 **FilterPanel: нет фильтра `radius`.**

Бэкенд поддерживает `radius` (по умолч. 1000, макс. 10000). FilterPanel не даёт пользователю изменить радиус поиска. Для приложения «найди туалет рядом» возможность расширить/сузить область поиска — это ключевой UX-элемент, особенно в малонаселённых районах (0 результатов при radius=1000, но есть при radius=3000).

**C2-2.6** 🟢 **`tokens.ts` утилита удалена из структуры проекта, но всё ещё упоминалась в v1.**

В v2 файл `src/utils/tokens.ts` убран из структуры — корректно, т.к. refresh теперь через cookie. Но стоит проверить, что нигде в документе нет остаточных ссылок на него. (Проверено — нет, чисто.)

**C2-2.7** 🟡 **Единый nginx.conf не обрабатывает CORS-заголовки при cookie auth.**

Строки 716-739: nginx.conf проксирует `/api/` на бэкенд, но не передаёт `Set-Cookie` настройки. Для httpOnly cookie с `SameSite=Strict` — это работает только при same-origin. При разработке (Vite proxy) — тоже ок. Но если фронтенд и API будут на разных доменах (напр. CDN для статики), cookie не будет отправляться. Стоит явно зафиксировать: фронтенд и API обязаны быть на одном домене.

### Линз 3: Допущения и риски

**C2-3.1** 🟡 **`react-spring-bottom-sheet` — библиотека с проблемами поддержки.**

Строка 141: выбран `react-spring-bottom-sheet`. Эта библиотека: (а) последний коммит был давно, (б) зависит от `react-spring`, которая активно развивается и может сломать совместимость, (в) имеет открытые issues по React 18 strict mode. Альтернативы: `vaul` (на Radix, активная разработка), кастомная реализация на CSS + pointer events. Стоит рассмотреть `vaul` — он уже совместим с shadcn/ui (который выбран в стеке).

**C2-3.2** 🟡 **openapi-typescript генерация — не описан процесс обновления.**

Строка 271-273: команда для генерации типов указана, но не описано: когда запускать? При каждом билде? Перед разработкой? В CI? В pre-commit hook? Без формализации типы быстро устареют.

**C2-3.3** 🟡 **Nominatim имеет rate limit: 1 запрос/сек.**

Строка 180: «Нужен геокодер: Nominatim». Nominatim free API имеет жёсткий rate limit — 1 запрос/сек, и требует User-Agent с контактным email. При массовом использовании это будет проблемой. Нужен либо: кэширование геокодирования на клиенте, либо собственный геокодер (Pelias, Photon), либо коммерческий API (Mapbox Geocoding).

**C2-3.4** 🟡 **Sentry Source maps: для Vite нужна специальная конфигурация.**

Строка 475: «Source maps загружаются в Sentry при CI-деплое». Но Vite по умолчанию не генерирует source maps в production. Нужен `build.sourcemap: true` в `vite.config.ts` + `sentry-cli` или `@sentry/vite-plugin`. Это не тривиальная настройка — стоит добавить в структуру проекта или в приоритеты.

**C2-3.5** 🟢 **shadcn/ui выбран, но не описана стратегия кастомизации.**

shadcn/ui — это копипаст-подход (компоненты копируются в проект). Как будет выглядеть тема? Цветовая палитра WTFApp? Это влияет на `globals.css` иTailwind-конфигурацию, но не описано.

### Линз 4: YAGNI & Раздувание scope

**C2-4.1** 🟡 **20 задач в Priority 1 — это очень большой MVP.**

Строки 748-768: 20 чекбоксов в MVP. Для одного разработчика это 2-3 месяца full-time работы. Если цель — быстрее запустить, стоит рассмотреть «MVP-0» из 8-10 задач: проект + API-клиент + auth + карта + BottomSheet + ToiletPage + NavBar. Без SearchPage, FilterPanel, ProfileEdit, VerifyEmail, PWA.

**C2-4.2** 🟡 **Playwright e2e в MVP — overengineering для одного разработчика.**

Строка 460-467: E2E-тесты полезны, но для одного разработчика на старте проекта они замедляют итерации. Стоит перенести в Priority 2, оставив в MVP только Vitest + Testing Library.

**C2-4.3** 🟢 **`useMediaQuery` хук присутствует в структуре, но нигде не используется.**

Строка 642: `useMediaQuery.ts` в хуках. Ни одна страница/компонент не упоминает media queries. Если нет адаптивных breakpoints за пределами Tailwind responsive — можно удалить.

**C2-4.4** 🟡 **iOS-подсказка «Добавьте на главный экран» — Priority не указан.**

Строка 374-376: описан баннер для iOS, но не указан приоритет. Вероятно, это Priority 2, но стоит явно зафиксировать.

### Линз 5: Техническая осуществимость

**C2-5.1** 🟡 **`vaul` (рекомендация) vs `react-spring-bottom-sheet`: технический риск выбора.**

(Дублёр C2-3.1.) Если остаётся `react-spring-bottom-sheet` — нужно явно протестировать на React 18 + strict mode до начала разработки. Если переход на `vaul` — он требует Radix (уже есть через shadcn/ui).

**C2-5.2** 🟡 **ky + httpOnly cookie: кросс-domain ограничения.**

ky по умолчанию не отправляет cookie при кросс-domain запросах. Если `VITE_API_URL` указывает на другой домен (не `/api/v1`), ky нужен `credentials: 'include'`. Строка 267: «ky отправляет cookie автоматически (same-origin через nginx proxy)» — верно для production, но для разработки нужно явно добавить `credentials: 'include'` в ky options, даже при Vite proxy. Стоит зафиксировать.

**C2-5.3** 🟡 **Cursor-пагинация в BottomSheet: нет описания UX «Загрузить ещё».**

Строка 400: `loadMore: () => void` — cursor-пагинация. Но как пользователь вызывает `loadMore`? Бесконечный скролл (IntersectionObserver)? Кнопка «Показать ещё»? Для BottomSheet с виртуализированным списком — это нетривиальная задача.

**C2-5.4** 🟢 **Сжатие фото через Canvas — Browser API-зависимость.**

Строка 207: «сжатие через Canvas перед отправкой». Это работает в современных браузерах, но Canvas.toBlob() для WebP не поддерживается в Safari < 16. Стоит явно указать: fallback на JPEG для неподдерживаемых браузеров.

**C2-5.5** 🟡 **`tailwind.config.ts` отсутствует в структуре проекта v2.**

Строка 685: структура содержит `vite.config.ts`, `tsconfig.json`, но нет `tailwind.config.ts`. В Tailwind CSS 4 конфигурация может быть через CSS, но если нужна кастомизация темы (цвета, шрифты), может понадобиться JS/TS конфиг. Стоит либо добавить, либо явно указать, что конфигурация через `globals.css`.

## Шаг 2 — Инверсия допущений

### Инверсия 1: Бэкенд не поддерживает httpOnly cookie

```
Допущение:  Бэкенд будет изменён для поддержки httpOnly cookie для refresh_token
Инверсия:   Бэкенд остаётся с возвратом refresh_token в JSON-ответе
Влияние:    Весь auth-поток фронтенда v2 неработоспособен.
            refresh_token невозможно установить как httpOnly cookie с фронтенда.
            Возврат к localStorage — единственный вариант, но это отбрасывает
            главное улучшение v2.
Исправление: Явно зафиксировать в бэкенд-дизайне: login/register должны
             устанавливать refresh_token как httpOnly cookie через Set-Cookie.
             Не возвращать refresh_token в JSON-ответе.
```

### Инверсия 2: Nominatim блокирует за rate limit

```
Допущение:  Nominatim free API будет доступен и достаточен
Инверсия:   Nominatim блокирует IP за превышение rate limit (1 req/sec)
Влияние:    Поиск по адресу перестаёт работать. Пользователь без геолокации
            не может пользоваться приложением.
Исправление: Кэшировать результаты геокодирования на клиенте (Map<query, result>).
             Debounce 1 сек на запросы к Nominatim.
             Fallback: при ошибке геокодирования → «Не удалось найти адрес.
             Попробуйте уточнить запрос.»
```

### Инверсия 3: react-spring-bottom-sheet не работает с React 18 strict mode

```
Допущение:  react-spring-bottom-sheet стабильно работает с React 18
Инверсия:   Библиотека имеет баги в React 18 strict mode (concurrent features)
Влияние:    BottomSheet — центральный элемент UX. Баги в нём = сломанное приложение.
Исправление: Заменить на `vaul` (активно поддерживается, Radix-based,
             совместим с shadcn/ui). Либо кастомная реализация на
             CSS scroll-snap + pointer events.
```
