## Шаг 1 — Пять линз критики

---

### Линз 1: Полнота (Completeness)

**C1.1** 🟡 **Нет стратегии типизации API-контракта.**
Документ упоминает «Валидация через Zod (схема = Pydantic-схема бэкенда)», но нет механизма синхронизации Pydantic-схем бэкенда и Zod-схем фронтенда. Ручное дублирование неизбежно разъедется. Нет упоминания `openapi-typescript` или генерации типов из OpenAPI-спецификации бэкенда.

**C1.2** 🟡 **Нет секции тестирования.**
В структуре проекта нет `__tests__/`, нет упоминания Vitest, Testing Library, Playwright. Для проекта на React + TypeScript это серьёзный пробел — кто и как будет проверять регрессии?

**C1.3** 🔴 **Нет стратегии accessibility (a11y).**
PWA для массового использования, но ни слова про ARIA, aria-label на кнопках карты, навигацию клавиатурой, контрастность. Для ToiletMarker и BottomSheet это особенно критично — Leaflet по умолчанию ужасен с точки зрения доступности.

**C1.4** 🟡 **Нет SEO / мета-тегов / Open Graph.**
Для PWA не критично, но если туалеты будут шариться по ссылкам (`/toilets/:id`), нужен SSR или хотя бы prerender для соцсетей. Сейчас SPA с `try_files $uri /index.html` — линк в Telegram/WhatsApp покажет пустую страницу.

**C1.5** 🟡 **Нет стратегии мониторинга и логирования.**
Нет Sentry, нет логирования клиентских ошибок. Как узнавать, что у пользователей падает приложение?

**C1.6** 🟢 **Нет явного описания loading-состояний для большинства страниц.**
Для MapPage описано (BottomSheet-состояния), но для ToiletPage, ProfilePage, SearchPage — что видит пользователь при загрузке? Skeleton? Spinner? Пустая страница?

**C1.7** 🟡 **Нет обработки email-верификации на фронтенде.**
Бэкенд имеет `POST /auth/verify-email`, но на фронтенде нет роута для обработки ссылки из email. `/verify-email?token=...` отсутствует в таблице роутов.

**C1.8** 🟡 **Нет обработки сброса пароля (reset-password).**
Есть `/forgot-password`, но бэкенд ожидает `POST /auth/reset-password` с токеном. На фронтенде нет роута `/reset-password?token=...` для приёма этого токена.

**C1.9** 🟡 **Нет UploadAvatar функционала.**
Бэкенд: `POST /api/v1/auth/me/avatar`. Frontend: ProfileEditPage описана абстрактно как «Редактировать профиль», но загрузка аватара не упоминается нигде — ни в компонентах, ни в структуре.

**C1.10** 🟢 **Нет упоминания CSRF-защиты.**
JWT в localStorage + SameSite cookies не упомянуты. Для API-only подхода с Bearer token это нормально, но стоит явно зафиксировать.

**C1.11** 🟡 **Нет описания конфигурации environment variables.**
Упомянуто `VITE_API_URL`, но нет `.env.example` с полным списком переменных, нет документации, какие переменные обязательны.

### Линз 2: Согласованность (Consistency)

**C2.1** 🔴 **Рассогласование роутинга с бэкенд-API.**
Frontend: `GET /nearby?lat=X&lon=Y`. Backend: `GET /api/v1/toilets/nearby?lat=X&lon=Y`. Это не ошибка самого документа (nginx проксирует), но в «Потоках пользователя» написано `GET /nearby?lat=X&lon=Y` — вводит в заблуждение, создаёт иллюзию что фронтенд зовёт корневой роут.

**C2.2** 🔴 **Фильтры FilterPanel не совпадают с API 1:1.**
Фильтр «Пол: Унисекс» в FilterPanel, но бэкенд имеет `gender = male | female | unisex | family`. В FilterPanel нет `family`. Также `paper_type` в фильтрах — `none / in_cabin / dispenser / both`, но в FilterPanel описан только `paper_type` как выпадающий список без уточнения значений.

**C2.3** 🟡 **`location_hint` vs `description` в бэкенде — путаница на фронтенде.**
Бэкенд: два поля `description` («Как найти внутри здания») и `location_hint` («Описание входа в здание»). Frontend AddToiletPage: только одно текстовое поле «Описание (как найти)». Куда уходит `location_hint`?

**C2.4** 🟡 **Удаление туалета: только soft delete, но UI описывает «Удалить».**
Frontend: «Кнопки "Редактировать" / "Удалить" (только для owner/moderator/admin)». Бэкенд: `DELETE /toilets/{id}` — soft delete (owner может), `DELETE /toilets/{id}/hard` — hard delete (только mod/admin). На UI не понятно, видит ли owner кнопку «Удалить» и что происходит при нажатии. Нужен ли confirm-диалог?

**C2.5** 🟡 **ProfilePage: `GET /api/v1/users/:id/toilets` — откуда фронтенд берёт `:id`?**
Написано `GET /api/v1/users/:id/toilets`, но это анонимный доступ. Для собственного профиля фронтенд должен использовать `GET /api/v1/auth/me` и затем подставить `user.id`. Или есть эндпоинт `/me/toilets`? В бэкенде его нет. Стоит явно описать цепочку.

**C2.6** 🟢 **React Router 7 — не уточнён режим.**
React Router 7 может работать как framework mode (Remix-подобный) и как library mode. Не указано, какой режим используется. С учётом Vite-сборки, скорее всего library mode, но стоит зафиксировать.

**C2.7** 🟡 **Zustand stores: `uiStore` упомянут в структуре, но нигде не описан.**
`stores/uiStore.ts` есть в дереве файлов, но нет ни одного упоминания его интерфейса или ответственности. Что он хранит? Состояние BottomSheet? Тосты? Тему?

### Линз 3: Допущения и риски (Assumptions & Risks)

**C3.1** 🔴 **«Токены в localStorage» — XSS-уязвимость.**
Документ: `accessToken + refreshToken → localStorage`. Это стандартный подход, но при любом XSS (через стороннюю библиотеку, через Leaflet плагин) токены доступны злоумышленнику. Альтернатива: httpOnly cookies для refresh token, и только access token в памяти (не в localStorage). Это серьёзное решение по безопасности, которое нужно принимать на этапе дизайна.

**C3.2** 🟡 **Leaflet + React — известные проблемы производительности.**
React-Leaflet — обёртка над императивным Leaflet. При большом количестве маркеров (1000+) без виртуализации DOM рендеринг будет тормозить. Кластеризация описана (`react-leaflet-cluster`), но нет лимита: что если `/nearby` вернёт 500 результатов? Какой上限 зум-уровень для показа всех маркеров без кластеров?

**C3.3** 🟡 **«Debounce 500ms при перемещении карты» — может быть слишком агрессивно.**
При быстром свайпе на мобильном + медленном соединении пользователь может видеть устаревшие маркеры. Нужен визуальный фидбек: показывать спиннер при обновлении? Затемнять маркеры?

**C3.4** 🟡 **Порядок загрузки: геолокация может занимать 10+ секунд.**
`watchPosition` на iOS может показать системный диалог → пользователь может отказаться → fallback к городу по умолчанию. Но «город по умолчанию» — что это? Москва? Определяется по IP? Если по IP — нужен отдельный API. Если нет — пользователь видит карту чужого города. Это критический UX-момент для приложения «найди туалет рядом».

**C3.5** 🟢 **Tailwind CSS 4 — ещё не стабильный на момент дизайна.**
Tailwind CSS v4 имел значительные API-изменения (CSS-first конфигурация). Если документ написан в мае 2026 — вероятно стабилизировался, но стоит явно указать `@tailwindcss/vite` как плагин вместо PostCSS.

**C3.6** 🟡 **Нет стратегии обработки истечения сессии при долгом неактивном использовании.**
Пользователь оставил PWA открытой на 2 дня. Access token истёк (15 мин), refresh token тоже (30 дней — ок, но что если 1 день?). При возврате → 401 → refresh → может не сработать → logout → потеря несохранённых данных (например, набранный отзыв).

### Линз 4: YAGNI & Раздувание scope (Scope Creep)

**C4.1** 🟡 **BottomSheet с тремя состояниями (COLLAPSED/HALF/FULL) — overengineering для MVP.**
Для MVP достаточно двух состояний: collapsed и expanded. HALF-состояние добавляет значительную сложность (snap-точки, анимации, жесты) ради показа 2-3 карточек. На мобильных экранах это особенно хрупко. Рекомендация: MVP — два состояния, HALF добавить в Priority 2.

**C4.2** 🟡 **PhotoUploader + PhotoGallery — полный функционал в MVP.**
Горизонтальный свайп, fullscreen-просмотр, превью при загрузке, валидация 10 фото, WebP — это значительный объём работы. Для MVP можно ограничиться 1-3 фото с простой загрузкой.

**C4.3** 🟢 **Headless UI + кастомные на Tailwind — избыточная зависимость?**
Headless UI даёт Listbox, Dialog, Switch — это используется в FilterPanel и формах. Но если использовать shadcn/ui (бесплатный, Tailwind-native, копипаст-компоненты), получится быстрее и без зависимости от отдельной библиотеки. Не критично, но стоит рассмотреть.

**C4.4** 🟡 **PWA offline-режим в MVP — спорно.**
«Базовый offline» включает: кэш `/nearby`, кэш `/toilets/:id`, баннер, скрытие кнопок, очередь действий. Это **не базовый** — это полноценный offline-first паттерн. «Базовый» — это просто Service Worker для статики. Очередь действий — это Priority 3.

**C4.5** 🟢 **Lucide React — отдельная зависимость для иконок.**
Минимальный оверхед, но для MVP можно использовать SVG inline или emoji для скорости. Не критично.

**C4.6** 🟡 **В структуре проекта 40+ файлов компонентов для MVP.**
Для первого релиза это огромный объём кода. Стоит явно указать, какие компоненты создаются в MVP, а какие — заглушки.

### Линз 5: Техническая осуществимость (Technical Feasibility)

**C5.1** 🔴 **Leaflet на мобильных — проблемы с жестами.**
Leaflet имеет известные проблемы с тач-жестами: конфликт свайпа BottomSheet со свайпом карты, pinch-zoom перехватывает scroll. Нет стратегии решения этого конфликта. React-Leaflet v4 + `leaflet` v1.9 имеют частичные фиксы, но это нужно явно тестировать и документировать.

**C5.2** 🟡 **vite-plugin-pwa + Workbox — тонкая настройка.**
Стратегия CacheFirst для OSM-тайлов на 7 дней может занять сотни мегабайт. Нужен лимит размера кэша (maxEntries / maxSize). Не указано.

**C5.3** 🟡 **«Мини-карта для выбора координат» в AddToiletPage.**
Встраивание второго Leaflet-экземпляра на страницу — это два тяжёлых DOM-дерева. На слабых мобильных устройствах это может быть проблемой. Альтернатива: использовать текущую позицию + поле ввода координат, или один MapView с переключением режима.

**C5.4** 🟢 **Zustand vs React Context — достаточность.**
Zustand — хороший выбор для глобального состояния (auth, map). Для MVP он достаточен. При росте сложности можно добавить TanStack Query для серверного состояния, но это не обязательно сейчас.

**C5.5** 🟡 **Нет стратегии code splitting.**
40+ компонентов, React Router, Leaflet — всё в одном бандле? Нужен lazy loading для роутов (React.lazy + Suspense). Особенно Leaflet (~200KB), который нужен только на MapPage и AddToiletPage.

## Шаг 2 — Инверсия допущений

### Инверсия 1: Безопасность токенов

```
Допущение:  localStorage — безопасное место для JWT-токенов
Инверсия:   XSS-атака через стороннюю зависимость компрометирует оба токена
Влияние:    Злоумышленник получает полный доступ к аккаунту от имени пользователя
Исправление: Refresh token — httpOnly cookie (set-cookie от бэкенда).
             Access token — хранить только в памяти (Zustand state, не persist).
             При перезагрузке страницы — refresh через cookie → новый access token.
```

### Инверсия 2: Геолокация всегда доступна

```
Допущение:  Geolocation API работает и пользователь даёт разрешение
Инверсия:   Пользователь отказывает в геолокации, или она недоступна (desktop без GPS)
Влияние:    Приложение «Где туалет?» не может выполнить свою основную функцию.
            Fallback «город по умолчанию» не решает задачу «найти ближайший».
Исправление: Добавить ручной ввод адреса/города как first-class flow.
             Показать «Введите адрес» prominently при отказе геолокации.
             Интегрировать Nominatim/геокодер для конвертации адреса → координаты.
```

### Инверсия 3: Мобильные браузеры поддерживают PWA

```
Допущение:  PWA полноценно работает на всех целевых платформах
Инверсия:   iOS Safari имеет ограничения: нет push-уведомлений, ограниченный Service Worker,
            7-дневный лимит на хранения IndexedDB, нет обновления SW без визита в Safari.
Влияние:    Ключевые PWA-фичи (offline, background sync) ненадёжны на iOS.
Исправление: Явно указать iOS как ограниченную платформу.
             Тестировать на iOS Safari отдельно.
             Рассмотреть «добавить на главный экран» подсказку как обязательную для iOS.
```

## Вердикт

```
VERDICT: 🟡 CONDITIONAL — исправить 5 блокеров, затем приступать к Design-to-Plan
```

**Обязательные исправления до реализации:**

1. **Безопасность токенов** — пересмотреть решение о localStorage для refresh token. Минимум — зафиксировать осознанный выбор с принятыми рисками.
2. **Конфликт жестов Leaflet + BottomSheet** — описать стратегию (touch-action CSS, gesture boundary, библиотека вроде `react-spring-bottom-sheet`).
3. **Согласовать FilterPanel с API** — добавить `family`, уточнить `paper_type`, убрать несуществующие фильтры.
4. **Добавить недостающие роуты** — `/verify-email`, `/reset-password`.
