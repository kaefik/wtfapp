## Шаг 1 — Пять линз критики

---

### Линз 1: Полнота (Completeness)

**B1.1** 🔴 **Нет секции тестирования — что, как, инструменты.**

Строка 432-437: структура `tests/` описана, но нет:
- Какой фреймворк тестирования? (pytest? unittest?)
- Как поднимать тестовую БД? (testcontainers? SQLite?)
- Как мокать Redis?
- Покрытие: минимальный %?
- CI-интеграция?

Для бэкенда на FastAPI это критический пробел — без чёткой стратегии тестирования geo-запросы (PostGIS) и auth-поток будут тестироваться вручную.

**B1.2** 🔴 **Нет описания ответа `GET /api/v1/toilets/{id}`.**

Для `/nearby` описан формат ответа (строки 303-327). Для `GET /toilets/{id}` — нет. Фронтенд рендерит ToiletPage с: название, адрес, этаж, все удобства, часы, description, location_hint, рейтинг, отзывы, фото. Какой именно JSON приходит? Это критический контракты для фронтенда.

**B1.3** 🟡 **Нет OpenAPI-спецификации / автоматической генерации.**

Фронтенд v3 использует `openapi-typescript` для генерации типов из OpenAPI-спецификации бэкенда. Бэкенд-дизайн упоминает FastAPI (автоматически генерирует OpenAPI), но:
- Не указан URL спецификации (`/openapi.json`? `/docs`?)
- Не описан процесс: генерируется автоматически? Валидируется в CI?
- Нет упоминания что фронтенд зависит от этой спецификации

**B1.4** 🟡 **Нет секции CI/CD.**

Нет описания:
- Как запускать тесты в CI
- Как деплоить
- Как запускать миграции при деплое
- Как откатывать миграции
- Docker registry

**B1.5** 🟡 **Нет стратегии логирования.**

FastAPI по умолчанию логирует в stdout, но:
- Какой формат логов? (JSON structued для production?)
- Какие уровни для каких модулей?
- Логирование медленных запросов?
- Корреляция request ID между фронтендом (Sentry) и бэкендом?

**B1.6** 🟡 **Нет мониторинга production.**

Нет упоминания:
- Health check описан (`/health`), но кто его вызывает? Docker? Kubernetes? Monit?
- Метрики: Prometheus? Grafana?
- Алерты при падении БД/Redis

**B1.7** 🟡 **Нет стратегии обработки upload файлов (multipart).**

Фронтенд v3 уточнил: один файл за запрос, `Content-Type: multipart/form-data`, поле `file`. Но бэкенд не описывает:
- Как FastAPI принимает файл (`UploadFile`?)
- Валидация: проверка magic bytes (а не только расширение)?
- Image processing: ресайз? Конвертация в WebP?
- Что происходит при превышении лимита фото (10 для туалета, 5 для отзыва)? 400? 409?

**B1.8** 🟡 **Нет описания `.env.example`.**

Строка 442: файл есть в структуре, но его содержимое не описано. Какие переменные обязательны? Значения по умолчанию? Для бэкенда критично: `DATABASE_URL`, `REDIS_URL`, `JWT_SECRET`, `CORS_ORIGINS`, `SMTP_*`.

**B1.9** 🟢 **Нет описания Pagination response format.**

Строка 218-220: «cursor-based пагинация», но не описан общий формат ответа. Для `/nearby` — есть пример. Для остальных — нет. Как выглядит `next_cursor` для `(created_at, id)` курсора?

**B1.10** 🟡 **Нет стратегии обработки изображений (ресайз, оптимизация).**

Фронтенд сжимает фото через Canvas, но бэкенд не описывает:
- Валидацию на стороне сервера (доверять клиенту нельзя)
- Ресайз на сервере (Pillow / sharp)?
- Генерация thumbnails?
- Максимальные размеры изображения

### Линз 2: Согласованность (с фронтенд-дизайном v3)

**B2.1** 🔴 **Auth: бэкенд возвращает refresh_token в JSON, фронтенд ожидает httpOnly cookie.**

Бэкенд (строка 259): `POST /api/v1/auth/login` — «Вход (возвращает access + refresh)».
Фронтенд v3: «Требования к бэкенду: login/register должны устанавливать refresh_token через Set-Cookie (httpOnly) вместо возврата в JSON-ответе».

Это **критическое рассогласование**, которое блокирует реализацию auth-потока фронтенда. Бэкенд-дизайн нужно обновить: login/register → `Set-Cookie` + JSON `{ access_token, user }` (без refresh_token).

**B2.2** 🔴 **Приоритеты не синхронизированы между бэкендом и фронтендом.**

| Функция | Бэкенд | Фронтенд | Конфликт |
|---------|--------|----------|----------|
| Email-верификация | P2 | P1 (VerifyEmailPage в MVP) | Фронтенд MVP не может работать без бэкенд-P2 |
| Password reset | P2 | P1 (ResetPasswordPage) | Аналогично |
| Часы работы + is_open_now | P2 | FilterPanel использует `is_open_now` фильтр в P1 | Фильтр не работает без бэкенд-P2 |
| Загрузка аватара | P1 (эндпоинт описан) | P2 (ProfileEditPage) | Ок, бэкенд раньше |
| Отзывы с фото | P1 | P1 (ReviewForm) | Ок, синхронно |

Фронтенд v3 MVP включает VerifyEmailPage, ResetPasswordPage, и фильтр `is_open_now` — но бэкенд относит эти функции к P2. Либо бэкенд должен поднять их в P1, либо фронтенд должен опустить в P2.

**B2.3** 🟡 **`/nearby` ответ: `lat`/`lon` как отдельные поля, но в БД `geom` — точка.**

Строка 310-312: ответ содержит `"lat": 55.75, "lon": 37.61`. В БД — `geom GEOMETRY(Point, 4326)`. Как бэкенд сериализует? Явно через `ST_Y(geom)` / `ST_X(geom)`? Стоит зафиксировать.

**B2.4** 🟡 **`POST /api/v1/toilets/` — не описан формат request body.**

Фронтенд отправляет: название, адрес, lat/lon, этаж, пол, тип, платный, цена, валюта, удобства (7 boolean), paper_type, description, location_hint. Бэкенд не описывает Pydantic-схему для создания туалета. Как передаются координаты: `lat`/`lon` отдельно? Или `geom` как WKT/GeoJSON?

**B2.5** 🟡 **`POST /api/v1/toilets/{id}/photos` — фото загружается до или после создания туалета?**

Проблема Chicken-and-egg: фронтенд отправляет форму создания туалета (`POST /toilets/`) → получает `{id}` → загружает фото (`POST /toilets/{id}/photos`). Но при редактировании — фото можно загружать сразу (туалет уже существует). В бэкенд-дизайне это не описано. Стоит явно указать: фото загружается только к существующему туалету.

**B2.6** 🟡 **`GET /api/v1/toilets/{id}` — «Детали туалета + отзывы». Но какие отзывы? Сколько?**

Строка 275: «Детали туалета + отзывы». Фронтенд показывает отзывы с пагинацией. Вложенные отзывы в ответе `GET /toilets/{id}` — первые 5? Или только ссылка на `GET /toilets/{id}/reviews/`? Если вложенные — может получиться огромный ответ для туалета с 1000 отзывов.

**B2.7** 🟡 **Роутер фотографий: `/toilets/photos/{id}` — нет в стандартном REST-паттерне.**

Строки 343, 345: `DELETE /api/v1/toilets/photos/{id}` и `DELETE /api/v1/reviews/photos/{id}`. Это sub-resource без parent ID в URL. Это нарушает REST-конвенцию и может конфликтовать с другими роутами. Альтернатива: `DELETE /api/v1/toilets/{toilet_id}/photos/{photo_id}`. Но текущий вариант тоже работает — просто стоит явно зафиксировать причину выбора.

### Линз 2.5: Внутренняя согласованность бэкенда

**B2.8** 🟡 **`is_deleted` фильтр не упомянут в API-параметрах `/nearby` и `/search`.**

Строка 78: `is_deleted = false` — soft delete. Строка 222: «Soft delete — удаление через is_deleted = true». Но в параметрах `/nearby` (строки 283-295) нет упоминания `is_deleted`. Предполагается, что бэкенд автоматически фильтрует `is_deleted = true`? Стоит явно указать.

**B2.9** 🟡 **`is_open_now` фильтр в `/nearby` требует вычисления — нет описания производительности.**

Строка 292: `is_open_now = true` — фильтр «показать только открытые». Строка 86: «вычисляется на лету из opening_hours. Кэшируется в Redis на 5 минут». Но для 50 туалетов в радиусе — нужно вычислить `is_open_now` для каждого. С кэшем — ок, но при cache miss (первый запрос, или 5 мин истекли) — 50 вычислений JSON parsing + time comparison. При 300 req/min это может стать узким местом.

**B2.10** 🟢 **`cabin_count` в модели туалета, но не в API-параметрах.**

Строка 75: поле `cabin_count` есть в БД. Но не упомянуто ни в параметрах `/nearby`, ни в ответе `/nearby`. Не критично, но стоит либо добавить в ответ, либо убрать из MVP.

### Линз 3: Допущения и риски

**B3.1** 🟡 **RS256 для JWT — нужен приватный ключ. Где он хранится?**

Строка 244: «JWT access-токен: RS256». RS256 требует RSA private key. Где он хранится? В `.env`? В Vault? Как ротируется (строка 246: «через версию ключа `kid»»)? При масштабировании на несколько инстансов — все инстансы должны иметь доступ к одному ключу.

**B3.2** 🟡 **Redis как single point of failure.**

Redis хранит refresh-токены, кэш `is_open_now`, rate limiting. Если Redis падает:
- Пользователи не могут авторизоваться (нет refresh)
- Rate limiting перестаёт работать
- `is_open_now` вычисляется на лету (медленнее, но работает)

Нет описания: Redis Sentinel? Redis Cluster? Persistence (AOF/RDB)? Для MVP — один инстант ok, но стоит зафиксировать как известный риск.

**B3.3** 🟡 **PostGIS `ST_DWithin` при высоком трафике.**

Гео-запрос с фильтрами + курсорная пагинация + вычисление расстояния — это сложный запрос. При 1000+ туалетов в Москве и 300 req/min:
- GiST индекс ускоряет пространственный поиск
- Но добавление 7+ WHERE условий (удобства) + `is_open_now` может замедлить
- Нет упоминания: `EXPLAIN ANALYZE` бенчмарки? PgHero? Query plan анализ?

**B3.4** 🟡 **«Локальная файловая система» для фото в MVP — проблема при деплое.**

Строка 369: «MVP: локальная файловая система». При docker-compose перезапуске контейнера — файлы теряются (если нет volume). При масштабировании на 2+ инстанса — файлы доступны только на одном. Стоит явно указать: Docker volume mount для `/media/photos/`.

**B3.5** 🟢 **bcrypt для хеширования паролей — ок, но не оптимально.**

Строка 36: «bcrypt». Для 2026 года argon2 — более современный выбор (GPU-resistant). Не критично, bcrypt адекватен.

**B3.6** 🟡 **Нет стратегии удаления/очистки файлов (orphaned photos).**

При удалении туалета (soft delete) — фото остаются в файловой системе. При hard delete — cascade удаляет записи из БД, но физические файлы? Кто их чистит? `storage.py` упоминается, но нет cron/задачи для очистки.

### Линз 4: YAGNI & Раздувание scope

**B4.1** 🟡 **`currency` поле (VARCHAR(3), DEFAULT 'RUB') — YAGNI для MVP.**

Строка 66: валюта. Если приложение изначально только для одного региона — зачем `currency`? Всегда RUB. Можно добавить в P2 когда появится международная поддержка.

**B4.2** 🟡 **`reports` сущность и эндпоинты — P2, но таблица описана детально в БД.**

Не критично — схема БД описана целиком, это нормально. Но API-эндпоинты reports (строки 347-353) можно было бы не описывать в MVP документе.

**B4.3** 🟢 **`locations` сущность (P2) — разумно отложена.**

Группировка туалетов в здания — P2. Верное решение, MVP с координатами в туалете достаточно.

**B4.4** 🟡 **`seed_osm.py` в MVP — спорно.**

Строка 431, строка 462: «Скрипт импорта демо-данных из OpenStreetMap». Это полезно для демонстрации, но для MVP можно просто иметь SQL-скрипт с 20 тестовыми туалетами. Импорт из OSM — нетривиальная задача (парсинг Overpass API, маппинг тегов, геометрия).

### Линз 5: Техническая осуществимость

**B5.1** 🟡 **Slowapi + async FastAPI — возможные проблемы.**

Строка 18: slowapi для rate limiting. slowapi синхронный (основан на limits + Flask-Limiter подходе). В async FastAPI это может блокировать event loop при каждом запросе. Альтернатива: custom middleware на Redis INCR + TTL (атомарный, async-friendly).

**B5.2** 🟡 **GeoAlchemy2 + async SQLAlchemy 2.0 — молодая связка.**

SQLAlchemy 2.0 async + GeoAlchemy2 — документация скудная. Есть известные issues с async sessions и гео-функциями. Стоит явно предусмотреть: spike/POC для проверки `ST_DWithin` + async session до начала реализации.

**B5.3** 🟡 **Cursor-пагинация на `(distance_m, id)` — нестандартная.**

Строка 295: base64-кодированный JSON `{"distance_m": 150.5, "id": "uuid"}`. Это сложнее стандартной cursor-пагинации на `(created_at, id)`. Реализация требует:
- Вычисление distance для каждой записи
- Сортировка по `(distance, id)`
- WHERE: `distance > cursor.distance OR (distance = cursor.distance AND id > cursor.id)`

Это рабочий подход, но стоит убедиться, что PostGIS + SQLAlchemy могут выразить этот запрос эффективно.

**B5.4** 🟢 **Trigger для пересчёта `rating_avg` / `reviews_count` — ok.**

Строка 79-80: триггер при INSERT/UPDATE/DELETE отзыва. Для PostgreSQL это стандартный и надёжный подход.

**B5.5** 🟡 **`opening_hours` JSONB + JSON Schema валидация — где?**

Строка 76: «Валидируется через JSON Schema». Но где именно? В Pydantic-схеме? В CHECK constraint БД? В application code? При INSERT/UPDATE — нужна валидация на уровне API.

## Шаг 2 — Инверсия допущений

### Инверсия 1: Redis недоступен

```
Допущение:  Redis всегда доступен и быстр
Инверсия:   Redis падает или перезагружается (AOF отключён, данные потеряны)
Влияние:    Все refresh-токены инвалидированы → все пользователи разлогинены.
            Rate limiting не работает → защита от DDoS отсутствует.
            is_open_now вычисляется без кэша → повышенная нагрузка на БД.
Исправление: Redis persistence: AOF включён (appendonly yes).
             Graceful degradation: если Redis недоступен, rate limiting пропускать
             (логировать warning), auth — только access token (без refresh).
```

### Инверсия 2: Фронтенд и бэкенд на разных доменах

```
Допущение:  Фронтенд и бэкенд всегда same-origin (nginx proxy)
Инверсия:   Фронтенд развёрнут на CDN (Netlify/Vercel), бэкенд на отдельном сервере
Влияние:    httpOnly cookie с SameSite=Strict не отправляется кросс-domain.
            Весь auth-поток ломается.
Исправление: Явно зафиксировать в бэкенд-дизайне: SameSite=Lax (не Strict)
             если возможен кросс-domain. Или: CORS + credentials: include.
             Но simplest: всегда same-origin (nginx proxy).
```

### Инверсия 3: Высокий трафик на `/nearby`

```
Допущение:  PostGIS + GiST справятся с ожидаемой нагрузкой
Инверсия:   1000 req/min на /nearby с 10 фильтрами → медленные запросы
Влияние:    Основной эндпоинт приложения тормозит. Пользователи видят пустую карту.
Исправление: Кэшировать /nearby в Redis (ключ = lat_lon_rounded + filters hash, TTL 30 сек).
             Для MVP — ok без кэша, но заложить место в архитектуре (services/toilet.py).
```
